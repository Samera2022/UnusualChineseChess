## 二、严重问题 (Critical)

### 2.1 [CRIT-01] 反射调用私有字段和方法 — [`GameStateImporter.java:84-95`](src/main/java/io/github/samera2022/chinese_chess/io/GameStateImporter.java:84)

**问题描述**：`importGameState()` 方法使用 Java 反射暴力访问 `GameEngine` 类的私有字段 `listeners`，并通过反射调用 `onMoveExecuted` 和 `onGameStateChanged` 方法。这是在规避类型安全的 API 设计。

**风险**：
- 字段名或方法签名变更 → 静默失败（catch 块吞掉异常）
- 违反 Java 模块系统封装原则
- 绕过正常的通知机制，可能导致监听器收到不一致的状态

**修复建议**：在 `GameEngine` 中提供一个公开方法（如 `notifyListenersForImport()`）专门处理导入后的通知：

```java
// 在 GameEngine 中新增：
public void notifyListenersAfterImport() {
    for (GameStateListener listener : listeners) {
        listener.onMoveExecuted(null);
        listener.onGameStateChanged(GameState.RUNNING);
    }
}
```

### 2.2 [CRIT-02] 反射回退到语义不同的方法 — [`ChineseChessFrame.java:385-391`](src/main/java/io/github/samera2022/chinese_chess/ui/ChineseChessFrame.java:385), [`ChineseChessFrame.java:410-416`](src/main/java/io/github/samera2022/chinese_chess/ui/ChineseChessFrame.java:410)

**问题描述**：`onForceMoveConfirm` 和 `onForceMoveApplied` 回调中，先尝试通过反射调用 `forceApplyMove()`，如果方法不存在则回退到 `makeMove()`。但这两个方法的行为完全不同：
- `forceApplyMove()` 跳过所有规则验证，强行移动棋子
- `makeMove()` 会进行回合检查、规则验证、棋子堆叠等完整逻辑

**风险**：如果 `forceApplyMove` 被重命名或移除，系统会静默切换到 `makeMove`，导致强制走子变成常规走子，可能被规则验证拒绝，造成双方状态不一致。

**修复建议**：直接调用 `forceApplyMove()` 而不使用反射。如果该方法存在签名问题，应该在编译时解决：

```java
// 替换反射调用为直接调用：
boolean applied = gameEngine.forceApplyMove(fromRow, fromCol, toRow, toCol, promotionType, selectedStackIndex);
```

### 2.3 [CRIT-03] 网络协议无加密 — [`NetworkSession.java`](src/main/java/io/github/samera2022/chinese_chess/net/NetworkSession.java)

**问题描述**：所有网络通信（MOVE、RESTART、SETTINGS、PING、JSON 帧）均通过纯文本 Socket 传输。虽然有基于 HMAC-SHA256 的签名机制用于防篡改，但数据本身不加密。

**风险**：
- 中间人可读取所有传输内容（走法、设置、PING 时间戳）
- Token 交换以明文进行，可被截获
- HMAC 密钥派生过于简单（直接字符串拼接），未使用 HKDF

**修复建议**：
- 如果需要安全性，考虑使用 TLS/SSL（SSLSocket）
- 对于局域网场景，添加一个明确的文档说明安全性假设
- 密钥派生改用标准 HKDF

### 3.1 [HIGH-01] 静态可变共享状态 — [`ChineseChessFrame.java:50,53,60`](src/main/java/io/github/samera2022/chinese_chess/ui/ChineseChessFrame.java:50)

```java
public static GameEngine gameEngine;       // line 50
public static MoveHistoryPanel moveHistoryPanel; // line 53
```

三个核心组件被声明为 `public static`，导致：

- 线程安全问题

**修复建议**：将 `gameEngine` 和 `moveHistoryPanel` 改为实例字段，`netController` 通过依赖注入传递：

```java
// 改为非静态
private GameEngine gameEngine;
private MoveHistoryPanel moveHistoryPanel;
private final NetModeController netController = new NetModeController();
```

### 3.2 [HIGH-02] 移动验证器与 UI 的循环依赖 — [`MoveValidator.java:80-82`](src/main/java/io/github/samera2022/chinese_chess/rules/MoveValidator.java:80)

**问题描述**：`MoveValidator.isValidMove()` 直接调用 `RuleSettingsPanel.isEnabled()` 和 `RuleSettingsPanel.getValue()` 读取 UI 组件状态：

```java
if (isEnabled(RuleRegistry.ALLOW_PIECE_STACKING.registryName) && 
    Integer.parseInt(RuleSettingsPanel.getValue(RuleRegistry.MAX_STACKING_COUNT.registryName)) > 1) {
```

这造成了 **规则引擎 ↔ UI 控件的硬循环依赖**。逻辑层不应该知道 UI 的存在。

**风险**：
- UI 未初始化时 `getValue()` 返回 `null` → `Integer.parseInt(null)` → NPE
- 规则引擎无法独立测试
- 违反了分层架构原则

**修复建议**：`MoveValidator` 已经持有 `rulesConfig` 引用，应直接从中读取：

```java
// 当前（错误）：
Integer.parseInt(RuleSettingsPanel.getValue(RuleRegistry.MAX_STACKING_COUNT.registryName))

// 修复后：
rulesConfig.getInt(RuleRegistry.MAX_STACKING_COUNT.registryName)
```

同样，`isEnabled()` 调用应替换为 `rulesConfig.getBoolean(registryName)`。

### 3.3 [HIGH-03] 撤销操作触发规则变更通知 — [`GameEngine.java:354-356`](src/main/java/io/github/samera2022/chinese_chess/engine/GameEngine.java:354)

**问题描述**：`undoLastMove()` 中调用 `rulesConfig.set()` 撤销规则变更，会触发 `GameRulesConfig.notifyRuleChange()`，进而可能触发 `ChineseChessFrame.rulesChangeListener`，导致网络同步、历史记录更新等一系列级联反应。

**风险**：
- 网络同步：撤销时的规则变更会通过 `sendSettingsSnapshotToClient` 发送给对端
- 规则变更记录：`rulesChangeListener` 中的 `addRuleChangeToHistory` 会在撤销时再次添加历史记录

**修复建议**：撤销规则变更时使用 `ChangeSource.UNDO`（已实现），并在 `rulesChangeListener` 中过滤掉 `UNDO` 来源的变更。

### 3.4 [HIGH-04] 重连/重放引擎状态恢复不完整 — [`GameStateImporter.java:101-160`](src/main/java/io/github/samera2022/chinese_chess/io/GameStateImporter.java:101)

**问题描述**：`rebuildInitialStateForReplay()` 通过"正向设置最终状态 → 反向撤销每一步"来反推初始状态。这个过程：
- 调用 `gameEngine.restart()` 清空所有状态
- 反复 `clearBoard()` / `pushToStack()`
- 在循环中调用 `undoLastMove()` 并触发规则变更

**风险**：
- 中途异常会导致状态不一致且无法恢复
- `undoLastMove()` 在撤销堆叠或俘虏走法时可能无法正确恢复
- 过于复杂，维护成本高

**修复建议**：改为直接从导入数据中构建初始棋盘状态，而非通过撤销反推。

### 3.5 [HIGH-05] `handleLeftClick` 条件逻辑过于复杂 — [`BoardPanel.java:241-258`](src/main/java/io/github/samera2022/chinese_chess/ui/BoardPanel.java:241)

**问题描述**：
```java
if ((!ChineseChessFrame.isNetSessionActive() && (piece == null || piece.isRed() != gameEngine.isRedTurn())) ||
    ChineseChessFrame.isNetSessionActive() && (piece == null || 
    !(localControlsRed == gameEngine.isRedTurn() && localControlsRed == piece.isRed())))
```

- `localControlsRed` 可能为 `null`（离线模式），比较 `null == gameEngine.isRedTurn()` 会产生自动装箱陷阱

```java
private boolean canSelectPiece(Piece piece) {

    if (!isNetSessionActive()) {
        return piece.isRed() == gameEngine.isRedTurn();
    }
    if (localControlsRed == null) return false;
    return localControlsRed == gameEngine.isRedTurn() && localControlsRed == piece.isRed();
```

### 3.6 [HIGH-06] 网络 `closeInternal()` 存在竞态条件 — [`NetworkSession.java:494-506`](src/main/java/io/github/samera2022/chinese_chess/net/NetworkSession.java:494)

**问题描述**：`closeInternal()` 可以被多个线程同时调用（服务端 accept 线程和客户端 IO 线程），对 `in`/`out`/`socket`/`serverSocket` 的 close 和设 null 操作没有同步保护。

**风险**：资源重复关闭、NullPointerException。

**修复建议**：添加 `synchronized` 保护或使用 `AtomicBoolean` 确保只关闭一次：

```java
private final AtomicBoolean closed = new AtomicBoolean(false);

private void closeInternal() {
    if (!closed.compareAndSet(false, true)) return;
    // ... 关闭资源
}
```

### 4.6 [MED-06] 马腿中间点计算在环绕模式下不正确 — [`MoveValidator.java:437-439`](src/main/java/io/github/samera2022/chinese_chess/rules/MoveValidator.java:437)
