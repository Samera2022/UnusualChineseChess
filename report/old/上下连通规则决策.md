# 上下连通 (Top-Bottom Connected) 规则实现决策文档

> **创建时间**: 2026-05-29
> **状态**: 已决策，待确认

## 一、规则概述

**"上下连通"**（`top_bottom_connected`）规则使棋盘上下边缘连通：row 0 的上方邻接 row 9，row 9 的下方邻接 row 0。棋盘形成垂直圆柱体（与已有的"左右连通"水平圆柱体正交）。

### 1.1 布局理解（概念展开视图）

标准棋盘 10×9，展开为概念性的 20 行（含 3 道楚河）：

```
一楚河一一一汉界一2    ← 冗余边界（row 0→9 的环绕）
兵一兵一兵一兵一兵     ← row 3 (黑兵)
一炮一一一一一炮一     ← row 2 (黑炮)
一一一一一一一一一      ← row 1
车马相士帅士相马车（红）← row 9→0 环绕→红方出现在黑方上方
一一一一一一一一一      ← row 8
一炮一一一一一炮一     ← row 7 (红炮)
兵一兵一兵一兵一兵     ← row 6 (红兵)
一一一一一一一一一
一楚河一一一汉界一1    ← 原始楚河 (row 4-5)
一一一一一一一一一
兵一兵一兵一兵一兵     ← row 6 (红兵)
一炮一一一一一炮一     ← row 7 (红炮)
一一一一一一一一一      ← row 8
车马相士帅士相马车（黑）← row 0 (黑方)
一一一一一一一一一      ← row 1
一炮一一一一一炮一     ← row 2 (黑炮)
兵一兵一兵一兵一兵     ← row 3 (黑兵)
一一一一一一一一一
一楚河一一一汉界一2    ← 冗余边界（row 9→0 的环绕）
```

**核心认知**：实际棋盘仍是 10×9，展开视图只是帮助理解环绕关系。

## 二、决策项

### D-1: 兵/卒的"过河"在环绕下如何判断？

**决策**：环绕不改变"过河"判断。红兵在 row≤4 时为已过河，黑卒在 row≥5 时为已过河。环绕仅影响移动可达性，不影响过河状态。

**理由**：环绕是空间拓扑变化，"过河"是规则状态变化，两者正交。若环绕即视为过河，则开局的兵/卒立即可横行，破坏平衡。

**替代方案**（已否决）：环绕即视为过河——过于激进。

---

### D-2: 王/将能否通过环绕出宫？

**决策**：不能。宫（Palace）的 3×3 范围限制仍然生效。环绕路径如果目的地仍在宫内则允许，否则拒绝。

**理由**：保持王/将的基础安全性。国际化王 (`international_king`) 不受宫限制，但仍受其本身规则约束。

---

### D-3: 飞将（王见王）如何适配环绕？

**决策**：垂直环绕路径计入飞将判断。如果两王在同一列且垂直环绕路径（含 wrap）无棋子阻挡，则触发飞将——允许直接吃王。

**理由**：与左右连通下的飞将逻辑对称。此行为与"取消对将" (`disable_facing_generals`) 互斥。

---

### D-4: 象/相是否能通过环绕过河？

**决策**：不能。象/相的"不能过河"限制 (`NO_RIVER_LIMIT` 关闭时) 仍然生效。环绕移动若导致跨河，则拒绝。

**理由**：保持象/相的传统限制。子规则 `top_bottom_connected_elephant` 开启后，环绕象的移动允许纵向环绕（但仍不能过河）。

---

### D-5: 马的绊脚在环绕下如何计算？

**决策**：子规则 `top_bottom_connected_horse` 开启时，马在纵向跨越 row 0↔9 时，绊脚位置按环绕方向计算。例如从 row 1 向上（→row 0→row 9）:"日"字形 col+1, rowDiff=2（1→9）时，绊脚在 row 0（中间行）。

**理由**：对称于左右连通马的绊脚逻辑。

---

### D-6: 车/炮的垂直环绕障碍计数

**决策**：仿照水平环绕的 `countHorizontalObstacles`，新增 `countVerticalObstacles`：
- 直接路径障碍数 + 环绕路径障碍数
- 选取障碍更少的路径为有效路径
- 车要求 0 障碍（任一路径），炮走子 0 障碍、吃子 1 障碍（炮架）

**理由**：对称于已有的左右连通逻辑，保持一致性。

---

### D-7: 士/仕在国际化模式下的垂直环绕

**决策**：`international_advisor` 开启时，士可以竖走/横走。若同时开启 `top_bottom_connected`，竖走允许环绕。

**理由**：与左右连通下的士横向环绕逻辑对称。

---

### D-8: 规则依赖与冲突

**决策**：

| 规则 | 类型 | 依赖 | 冲突 |
|------|------|------|------|
| `top_bottom_connected` | CHECK_BOX | 无 | `left_right_connected`（可共存但暂不推荐同时开启，行为未定义） |
| `top_bottom_connected_horse` | CHECK_BOX | `top_bottom_connected` | 无 |
| `top_bottom_connected_elephant` | CHECK_BOX | `top_bottom_connected` | 无 |

**理由**：结构与 `left_right_connected` 系列一致。上下+左右同时开启形成"环面棋盘"（torus），行为复杂度高，暂标记为谨慎使用。

---

### D-9: 将死检测适配

**决策**：`CheckDetector` 使用的 `MoveValidator.isValidMove()` 已经通过规则配置生效，无需额外改动。

**理由**：`CheckDetector.isInCheck()` 遍历所有敌方棋子调用 `validator.isValidMove()`，后者会根据 `top_bottom_connected` 自动计算环绕路径。`canEscapeCheck()` 亦然。

---

### D-10: 棋盘渲染

**决策**：在棋盘上下边缘添加视觉提示（虚线或颜色渐变），表示"此处连通"。不改变棋盘尺寸。

**理由**：帮助玩家理解环绕规则，类似左右连通已有的视觉处理。
