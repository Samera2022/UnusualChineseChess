# UnusualChineseChess 构建系统分析报告

## 当前构建系统状况

### 1. 构建系统概览

项目目前存在**多重构建系统并存**的情况，主要包括：

- **Maven构建系统** (正在迁移中)
- **PowerShell构建脚本** (传统构建方式)
- **GitHub Actions CI/CD** (自动化发布)
- **IDE项目文件** (IntelliJ IDEA)

### 2. 各构建组件分析

#### 2.1 Maven构建系统 ([`pom.xml`](pom.xml))

**当前状态**: 迁移进行中

**配置分析**:
```xml
<!-- 基本信息 -->
<groupId>io.github.samera2022</groupId>
<artifactId>UnusualChineseChess</artifactId>
<version>2.1.0</version>

<!-- 依赖管理 -->
<dependencies>
    <dependency>
        <groupId>com.google.code.gson</groupId>
        <artifactId>gson</artifactId>
        <version>2.11.0</version>
    </dependency>
</dependencies>

<!-- 构建插件 -->
<plugins>
    <maven-compiler-plugin> <!-- 编译配置 -->
    <maven-assembly-plugin> <!-- 打包配置 -->
</plugins>
```

**问题分析**:
- ✅ 基础配置完整
- ⚠️ 依赖管理不完整（缺少其他jar依赖）
- ⚠️ 构建目标配置需要完善

#### 2.2 PowerShell构建脚本 ([`.build/build.ps1`](.build/build.ps1))

**功能**: 多格式打包（JAR、ZIP、EXE）

**构建流程**:
```
1. 读取配置 (properties.json)
2. 生成自定义JRE (jlink)
3. 根据类型执行打包:
   - jar: 复制Maven构建的JAR文件
   - zip: 使用jpackage创建应用镜像并压缩
   - exe: 使用jpackage + Enigma Virtual Box打包
```

**技术栈**:
- **jlink**: 创建精简JRE
- **jpackage**: Java应用打包工具
- **Enigma Virtual Box**: 虚拟化打包工具
- **WiX Toolset**: Windows安装包工具

**问题分析**:
- ✅ 功能完整，支持多种发布格式
- ⚠️ 依赖外部工具（Enigma Virtual Box）
- ⚠️ Windows平台限制
- ⚠️ 与Maven构建流程重复

#### 2.3 GitHub Actions CI/CD ([`.github/workflows/official-release.yml`](.github/workflows/official-release.yml))

**触发条件**: 标签推送

**构建流程**:
```
1. 环境设置 (JDK 11, WiX, Enigma Virtual Box)
2. Maven构建 (mvn clean package)
3. 多格式打包 (调用build.ps1)
4. 生成更新日志
5. 发布到GitHub Releases
```

**问题分析**:
- ✅ 自动化程度高
- ✅ 支持多格式发布
- ⚠️ 仅支持Windows平台
- ⚠️ 依赖外部工具安装

#### 2.4 其他构建相关文件

- **`rerun.ps1`**: 标签重新推送脚本
- **`TO-DO.txt`**: 开发待办事项
- **`UnusualChineseChess.iml`**: IntelliJ IDEA项目文件
- **`.build/properties.json`**: 构建配置

## 构建系统问题分析

### 1. 主要问题

#### 1.1 系统复杂度高
- **多重构建路径**: Maven、PowerShell、GitHub Actions
- **工具依赖多**: jlink、jpackage、Enigma Virtual Box、WiX
- **平台限制**: 主要针对Windows环境

#### 1.2 依赖管理混乱
- **Maven依赖不完整**: 只声明了Gson，其他依赖缺失
- **外部工具依赖**: 需要手动安装多个工具
- **版本管理**: 依赖版本未统一管理

#### 1.3 构建流程冗余
- **重复打包**: Maven和PowerScript都进行打包
- **资源管理**: 资源文件路径不一致
- **配置分散**: 构建配置分布在多个文件中

### 2. 具体技术问题

#### 2.1 Maven配置问题
```xml
<!-- 当前问题 -->

    <!-- 只声明了gson，其他依赖缺失 -->
</dependencies>

<!-- 需要添加的依赖 -->
<!-- 其他项目依赖的jar文件 -->
```

#### 2.2 构建脚本问题
```powershell

    # Enigma Virtual Box路径硬编码
```

## 改进方案

### 1. 短期改进 (1-2周)

#### 1.1 完善Maven配置
```xml
<!-- 添加缺失的依赖 -->
<dependencies>
    <dependency>
        <groupId>com.google.code.gson</groupId>
        <artifactId>gson</artifactId>
        <version>2.11.0</version>
    </dependency>
    <!-- 添加其他必要的依赖 -->
</dependencies>

<!-- 统一构建配置 -->
<build>
    <plugins>
        <maven-compiler-plugin>
            <configuration>
                <source>11</source>
                <target>11</target>
            </configuration>
        </maven-compiler-plugin>
        <maven-assembly-plugin>
            <configuration>
                <descriptorRefs>
                    <descriptorRef>jar-with-dependencies</descriptorRef>
                </descriptorRefs>
            </configuration>
        </maven-assembly-plugin>
    </plugins>
</build>
```

#### 1.2 简化构建流程
- **统一使用Maven进行基础构建**
- **保留PowerScript用于高级打包**
- **清理冗余的构建文件**

### 2. 中期改进 (1-2个月)

## 具体实施步骤

### 阶段一：Maven配置完善
1. **分析现有依赖**: 检查lib目录和代码中的依赖
2. **添加Maven依赖**: 将所有依赖声明到pom.xml
3. **测试Maven构建**: 确保`mvn clean package`正常工作
4. **验证功能**: 确保构建产物功能正常

1. **统一资源管理**: 将所有资源文件移动到标准目录
2. **清理冗余文件**: 删除不必要的构建脚本和配置
3. **文档更新**: 更新构建说明文档

### 阶段三：跨平台支持
1. **Linux/macOS构建测试**: 验证在其他平台的构建
2. **GitHub Actions多平台配置**: 添加多平台构建任务
3. **打包工具标准化**: 使用跨平台兼容的工具

## 风险评估

### 技术风险
- **依赖冲突**: 添加Maven依赖可能引起版本冲突
- **构建失败**: 构建流程变更可能导致临时构建失败
- **功能回归**: 构建系统变更可能影响现有功能

### 缓解措施
- **分阶段实施**: 逐步迁移，避免大规模变更
- **充分测试**: 每个阶段都进行充分测试
- **回滚计划**: 准备快速回滚方案
