## 1. 当前架构审计

### 1.1 项目概览

当前项目是 **单一 Maven 模块**（[`pom.xml`](pom.xml:1)），所有代码位于同一个 `src/main/java/io/github/samera2022/chinese_chess/` 包树下。共 31 个 Java 源文件，仅依赖 Gson 一个外部库。

| 问题 | 严重程度 | 涉及文件 | 说明 |
|------|----------|----------|------|

## 3. ucc-common 解耦评估

### 3.1 评估结论：**需要解耦，且应作为最高优先级**

`ucc-common` 模块是其余三个模块能够独立编译和运行的基础。以下是详细分析：

### 3.2 应纳入 ucc-common 的类

| 类 | 理由 | 外部依赖 | 风险 |
|---|---|---|---|
| **Piece** + `Piece.Type` 枚举 | 被 engine/ui/net/io 全部使用，是核心数据类型 | 无 | 无 |
| **Move** | 被 engine/ui/io 使用，着法数据结构 | 仅依赖 Piece | 需注意 Gson 序列化（无参构造） |
| **HistoryItem** | 接口，被 Move 和 RuleChangeRecord 实现 | 无 | 无 |
| **RuleChangeRecord** | 规则变更记录，被 engine/ui/io 使用 | 无（有无参构造给 Gson） | 注意 `HIDDEN_KEYS` 静态集合 |
| **Consts** | 被 RuleRegistry 和 RuleSettingsPanel 使用 | 无 | 2个常量，极其简单 |
| **GameConfig** | UI 尺寸/颜色/棋盘参数常量，被 UI 和 engine 使用 | 无（纯静态常量） | 无 |
| **RuleRegistry** 枚举 | 规则元数据契约，被所有模块使用 | 仅依赖 Consts | 26条规则枚举，是系统的"配置契约" |
| **UpdateInfo** | 版本信息，被 UI 和 net 使用 | 依赖 Gson + classpath 资源 | 静态初始化块会读取 `updates.json` |

### 3.3 不应纳入 ucc-common 的类

| 类 | 应归属 | 理由 |
|---|---|---|
| **GameRulesConfig** | ucc-core | 包含运行时状态管理、线程池、监听器机制 |
| **RulesConfigProvider** | ucc-core | 全局单例提供者，管理 GameRulesConfig 生命周期 |
| **MoveValidator** | ucc-core | 纯引擎逻辑，依赖 Board |
| **CheckDetector** | ucc-core | 纯引擎逻辑，依赖 Board + MoveValidator |
| **Board** | ucc-core | 棋盘状态管理，引擎核心 |
| **GameEngine** | ucc-core | 游戏逻辑中枢 |

### 3.4 RuleRegistry 特殊分析

[`RuleRegistry`](src/main/java/io/github/samera2022/chinese_chess/rules/RuleRegistry.java:7) 是一个包含 26 条规则的枚举。它既是**编译期契约**（定义了规则名称、默认值、依赖/冲突关系），也是**运行时引用**（`registryName` 字符串被各模块硬编码使用）。

**当前使用方式（脆弱）**：
```java
// 在 GameEngine 中
rulesConfig.getBoolean(RuleRegistry.TOP_BOTTOM_CONNECTED.registryName)
// 在 MoveValidator 中
rulesConfig.getBoolean(RuleRegistry.ALLOW_PIECE_STACKING.registryName)
// 在 ChineseChessFrame 中
if (RuleRegistry.TOP_BOTTOM_CONNECTED.registryName.equals(key))
```

这种模式意味着 RuleRegistry 必须对所有模块可见。放入 ucc-common 是最合理的选择——它本质上是系统的"规则 ID 常量表"。

## 4. 目标模块架构

### 4.1 Maven 模块结构

```
UnusualChineseChess (parent POM, packaging=pom)
├── ucc-common/          ← 共享契约层
├── ucc-core/            ← 后端引擎层
├── ucc-app/             ← 前端 UI 层
└── ucc-api/             ← API/网络层
```

### 4.2 模块职责与边界

```
┌──────────────────────────────────────────────────────────┐
│                      ucc-app (UI)                        │
│  ChineseChessFrame, BoardPanel, InfoSidePanel,           │
│  RuleSettingsPanel, MoveHistoryPanel, AboutDialog,       │
│  UpdateInfoDialog, DocumentInputFilter                   │
│                                                          │
│  依赖: ucc-core, ucc-api, ucc-common, Swing, Gson       │
├──────────────────────────────────────────────────────────┤
│                      ucc-api (API)                       │
│  NetworkSession, NetModeController,                      │
│  GameStateExporter, GameStateImporter                    │
│                                                          │
│  依赖: ucc-common, Gson                                  │
│  ⚠ 不依赖 ucc-core! (通过接口交互)                        │
├──────────────────────────────────────────────────────────┤
│                      ucc-core (Engine)                   │
│  Board, GameEngine, MoveValidator, CheckDetector,        │
│  GameRulesConfig, RulesConfigProvider                    │
│                                                          │
│  依赖: ucc-common, Gson                                  │
│  ⚠ 不依赖 ucc-app, 不依赖 ucc-api                        │
├──────────────────────────────────────────────────────────┤
│                     ucc-common (Shared)                  │
│  Piece, Move, HistoryItem, RuleChangeRecord,             │
│  Consts, GameConfig, RuleRegistry, UpdateInfo            │
│                                                          │
│  依赖: Gson (仅 UpdateInfo 需要)                          │
│  ⚠ 不依赖任何其他模块                                     │
└──────────────────────────────────────────────────────────┘
```

### 4.3 依赖方向规则

```
ucc-app ──────→ ucc-core ──────→ ucc-common
   │               │                  ↑
   │               └──→ ucc-api ──────┘
   │                              ↑
   └──────────→ ucc-api ─────────┘
```

- ✅ **ucc-common** 不依赖任何业务模块
- ✅ **ucc-api** 不依赖 ucc-core（关键约束，避免循环依赖）
- ✅ **ucc-core** 依赖 ucc-common + ucc-api（调用 GameStateSerializer 等序列化工具）
- ✅ **ucc-app** 依赖 ucc-core + ucc-api + ucc-common

> **修正说明**: 原方案规定 ucc-core 不得依赖 ucc-api。经审查，`ucc-core → ucc-api → ucc-common` 是纯单向依赖，不产生循环。真正的约束是 ucc-api 不得依赖 ucc-core（否则形成 `ucc-api ← ucc-core → ucc-api` 循环）。GameEngine 调用 GameStateSerializer 是自然的代码组织，强行将序列化工具搬入 ucc-core 属于教条式设计。

### 4.4 关键接口抽象

当前最大的耦合痛点是 `ucc-api` 中的 `GameStateExporter`/`GameStateImporter` 和 `NetworkSession` 直接依赖具体类 `GameEngine` / `Board`。解耦后需引入以下接口（定义在 ucc-common）：

```java
// ucc-common 中定义接口
public interface GameStateProvider {
    BoardState getBoardState();
    boolean isRedTurn();
    GameStatus getGameStatus();
    List<Move> getMoveHistory();
    List<RuleChangeRecord> getRuleChangeHistory();
    JsonObject getRulesSnapshot();
}

public interface GameStateReceiver {
    void applyBoardState(BoardState state);
    void setRedTurn(boolean isRedTurn);
    void applyRulesSnapshot(JsonObject snapshot);
    void addMoveToHistory(Move move);
    void addRuleChangeToHistory(RuleChangeRecord record);
}
```

### 5.3 Phase 3: 创建 ucc-api

**目标**：网络通信和 IO 操作独立为 API 模块。

**迁移类列表**：

| 源路径 | 目标路径 | 改动 |
|---|---|---|
| `net/NetworkSession.java` | `ucc-api/.../api/net/NetworkSession.java` | 更新 import Piece, UpdateInfo |
| `net/NetModeController.java` | `ucc-api/.../api/net/NetModeController.java` | 更新 import |

| `io/GameStateImporter.java` | `ucc-api/.../api/io/GameStateImporter.java` | **需重构：通过接口而非具体类访问 GameEngine** |
| `filter/DocumentInputFilter.java` | `ucc-api/.../api/filter/DocumentInputFilter.java` | 或移到 ucc-app（实际只被 UI 使用） |

**关键重构：IO 类解耦**

[`GameStateExporter`](src/main/java/io/github/samera2022/chinese_chess/io/GameStateExporter.java:35) 和 [`GameStateImporter`](src/main/java/io/github/samera2022/chinese_chess/io/GameStateImporter.java:34) 当前直接接收 `GameEngine` 并调用其内部方法。重构方案：

**方案 A（推荐）**：定义接口在 ucc-common，ucc-core 实现
```java
// ucc-common
public interface GameStateAccessor {
    BoardState getBoardState();
    boolean isRedTurn();
    GameEngine.GameState getGameState(); // 或将 GameState 枚举移至 common
    List<Move> getMoveHistory();
    List<RuleChangeRecord> getRuleChangeHistory();
    JsonObject getRulesSnapshot();
    void clearMoveHistory();
    void clearRuleChangeHistory();
    void addMoveToHistory(Move move);
    void addRuleChangeToHistory(RuleChangeRecord record);
    void setRedTurn(boolean isRedTurn);
    void restart();
    void saveInitialStateForReplay();
}
```

**方案 B**（最小改动）：ucc-api 依赖 ucc-core（不推荐，破坏单向依赖）

**推荐方案 A**，但需将 `GameEngine.GameState` 枚举提升到 ucc-common。

**NetworkSession 改动**：
- [`NetworkSession`](src/main/java/io/github/samera2022/chinese_chess/net/NetworkSession.java:22) 对 `Piece` 的 import 实际上仅用于未使用的字段，可以移除
- 对 `UpdateInfo` 的依赖（获取版本号）保留，因为 `UpdateInfo` 已在 ucc-common

**pom.xml**：
```xml
<!-- ucc-api/pom.xml -->
<dependencies>

        <groupId>io.github.samera2022</groupId>

</dependencies>
```

**迁移类列表**：

## 7. 风险评估与缓解

### 7.1 高风险项

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| **GameEngine 拆分导致行为回归** | 高 | 当前无单元测试。Phase 2 前先为 `makeMove()`, `undoLastMove()`, `isValidMove()` 编写特征测试（Characterization Tests） |
| **规则一致性检查 (enforceRuleConsistency) 失效** | 高 | 该逻辑在 `GameRulesConfig` 内部，模块拆分不涉及改动，仅更新 import。风险可控 |
| **NetworkSession HMAC 签名验证** | 中 | 仅更新 import（`UpdateInfo`）。保持不变 |
| **包名变更导致序列化兼容性** | 中 | Gson 默认不使用全限定类名。`Move`, `Piece.Type`, `RuleChangeRecord` 均使用字段名序列化，不受包名影响 |
| **updates.json 资源路径** | 低 | 从根模块移至 ucc-common，`UpdateInfo.class.getResourceAsStream("/updates.json")` 依然有效 |

### 7.2 中风险项

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| **ChineseChessFrame 拆分引入协调 Bug** | 中 | 分步拆分：先提取内部类 → 验证 → 提取独立类。每次提取后手动回归测试 |
| **RulesConfigProvider 单例跨模块** | 中 | 单例保持在 ucc-core。ucc-app 通过依赖注入获取引用（已在构造函数中传递） |
| **GameRulesConfig 线程池生命周期** | 低 | `shutdown()` 方法保持不变。在 ucc-app 的 `WindowListener` 中调用 |

### 7.3 低风险项

- `Consts`, `GameConfig` 纯常量类移动
- `Piece`, `Move`, `HistoryItem`, `RuleChangeRecord` 纯数据类移动
- `DocumentInputFilter` 移动（仅被 UI 文本字段使用）
- `AboutDialog`, `UpdateInfoDialog` 简单对话框移动

```java
    boolean requestUndo(String playerId);

    void resign(String playerId);

    /** 获取同步快照（用于新客户端加入/断线重连） */
    String getSyncSnapshotJson();

    /** 该会话是否已结束 */
```
