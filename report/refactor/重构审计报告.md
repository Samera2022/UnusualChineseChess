## 🔴 阻塞性问题

### 1. ucc-api 中 NetworkSession 残留未使用的 Swing/AWT import（破坏 Android 兼容性）

**位置**: [`NetworkSession.java`](ucc-api/src/main/java/io/github/samera2022/chinese_chess/api/net/NetworkSession.java:7-9)

```java
import javax.swing.SwingUtilities;  // 🔴 Android 不可用
import java.awt.Component;          // 🔴 Android 不可用
import java.awt.Container;          // 🔴 Android 不可用
```

**影响**: `ucc-api` 的定位是平台无关的 API 层，供 `ucc-app`（Swing）和未来 `ucc-android` 共同使用。引入了 Swing/AWT 后，Android 客户端无法使用此模块，必须 fork 或重写整个网络层。

**修复方案**: 搜索 `NetworkSession.java` 中对 Swing/AWT 的使用，发现仅在一处使用了 `SwingUtilities.invokeLater()` 来回调 Listener。应改为由调用方决定线程策略：

```java
// 当前代码（在 handleLine 中）
if (listener != null) {
    // 不需要在这里 invokeLater，由 Listener 实现方自己决定
    listener.onPeerMove(fr, fc, tr, tc, selectedStackIndex);
}

// 修复：
// 1. 移除 NetworkSession 中的所有 Swing/AWT import
// 2. 移除可能存在的 SwingUtilities.invokeLater 包装
// 3. 让 Listener 的实现方（如 ucc-app 中的 NetGameCoordinator）自行包装 SwingUtilities.invokeLater
```

注：经检查，`NetworkSession.java` 第 7-9 行的 import 实际上在当前代码中**未被使用**（`SwingUtilities`/`Component`/`Container` 在代码体中未出现）。可以直接删除这 3 个 import 行。

---

> **关于"ucc-core 不应依赖 ucc-api"的更正**: 经重新审查，原始方案中"ucc-core 只依赖 ucc-common"是一个过度约束。`ucc-core → ucc-api → ucc-common` 是完全单向的，不产生循环——真正的约束是 ucc-api 不得依赖 ucc-core（否则形成循环）。`GameEngine` 调用 `GameStateSerializer` 是自然的代码组织。**此项已从阻塞性问题列表中移除**，同时更新架构文档中的依赖规则。

### 3. GameEngine 内仍保留重复的内部枚举和接口

**位置**: [`GameEngine.java`](ucc-core/src/main/java/io/github/samera2022/chinese_chess/core/engine/GameEngine.java:56-66)

```java
// 第 56-61 行：内部 GameState 枚举（与 common.GameStatus 重复）
public enum GameState {
    RUNNING, RED_CHECKMATE, BLACK_CHECKMATE, DRAW
}

// 第 63-66 行：内部 GameStateListener 接口（与 common.GameStateListener 重复）
public interface GameStateListener {
    void onGameStateChanged(GameState newState);
    void onMoveExecuted(Move move);
}
```

**影响**: `common.GameStatus` 和 `common.GameStateListener` 已经定义在 ucc-common 中，并且 GameEngine 已通过包装器（`commonListenerWrappers`）实现了 `GameStateAccessor` 接口的双向转换。内部枚举和接口保留是为了向后兼容 `ucc-app` 中的现有监听器。

**建议**: 在后续迭代中逐步迁移 `ucc-app` 的 `GameController` 和 `MoveHistoryPanel` 从 `GameEngine.GameStateListener` → `common.GameStateListener`，然后删除内部重复类型。

### 4. BoardPanel 构造函数签名变化未完全消除 NetModeController 耦合

**位置**: [`BoardPanel.java`](ucc-app/src/main/java/io/github/samera2022/chinese_chess/app/ui/BoardPanel.java:28-30)

```java
private final BooleanSupplier isNetSessionActive;
private final Consumer<JsonObject> boardSetupSender;
private final Supplier<Boolean> localControlsRedSupplier;
```

✅ 已消除 `getNetController()` 向上穿透。但 `boardSetupSender` 的类型是 `Consumer<JsonObject>`，仍然依赖 Gson 的 `JsonObject`。对于纯 UI 面板来说这是可接受的，但如果未来 Android 端想复用 `BoardPanel`（不太可能因为是 Swing 类），则需进一步抽象。

**评估**: 当前可接受，因为 `BoardPanel` 本身就是 `javax.swing.JPanel` 的子类，天然绑定 Swing。

## 已纠正的误判

### ~~原 🔴#1: ucc-core 非法依赖 ucc-api~~ → 撤销

经重新审查，`ucc-core → ucc-api` 的依赖是**合理且正确的**：

- 依赖链 `ucc-core → ucc-api → ucc-common` 是纯单向的，不产生循环
- 真正的架构约束是 **ucc-api 不得依赖 ucc-core**（否则形成循环），当前满足
- `GameEngine` 调用 `GameStateSerializer` 是自然的代码组织方式
- 对三个未来模块（android/ai/server）均无负面影响

**已纠正**: `ucc-core/pom.xml` 恢复了 `ucc-api` 依赖，`mvn clean compile` 通过。
