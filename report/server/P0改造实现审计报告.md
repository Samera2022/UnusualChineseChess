#### ⚠️ 发现的问题

**🔴 问题 1: `isValidMove()` 未加锁** ([:268-284](../../ucc-core/src/main/java/io/github/samera2022/chinese_chess/core/engine/GameEngine.java:268))

```java
// 当前代码 — 无锁保护
public boolean isValidMove(int fromRow, int fromCol, int toRow, int toCol, int selectedStackIndex) {
    if (gameState != GameStatus.RUNNING) return false;      // 读取 gameState 无锁
    List<Piece> fromStack = board.getStack(fromRow, fromCol); // 读取 board 无锁
    ...
    if (piece == null || piece.isRed() != isRedTurn) ...      // 读取 isRedTurn 无锁
    return validator.isValidMove(...);
}
```

**风险**: 多线程场景下，`isValidMove()` 读取的 `gameState`、`isRedTurn`、`board` 状态可能与其他写操作不一致。例如：读到一个正在进行 `undoLastMove()` 的中间状态的棋盘。

**影响**: 低至中。`isValidMove()` 通常用于 UI 提示（非关键路径），但在服务端可能用于着法预校验。

**建议**: 添加 `engineLock.lock()` 包裹，或标注 `@NotThreadSafe` 并在文档中说明此方法应在持有锁的上下文中调用。

**🟡 问题 2: `getBoard()` 返回可变引用** ([:497-504](../../ucc-core/src/main/java/io/github/samera2022/chinese_chess/core/engine/GameEngine.java:497))

```java
public Board getBoard() {
    engineLock.lock();
    try {
        return board;  // 直接返回内部可变对象
    } finally {
        engineLock.unlock();
    }
}
```

**风险**: 调用方获得 `Board` 引用后，可以在**无锁**的情况下直接修改棋盘状态（如 `board.pushToStack()`），绕过 `GameEngine` 的所有线程安全保护。

**影响**: 中。当前调用方（GUI、测试）通常遵循"拿完就用"模式，但在 ucc-server 场景中，如果 Worker 线程获取 `board` 引用后跨线程传递，会导致并发修改。

**建议**: 
- 方案 A（激进）: 返回 `board.toState()` 不可变快照
- 方案 B（务实）: 标注 `@Deprecated` 并注释警告，新增 `getBoardSnapshot()` 方法

**🟡 问题 3: `getPiece()` / `getStack()` / `getStackSize()` / `getBoardRows()` / `getBoardCols()` 无锁** ([:509-531](../../ucc-core/src/main/java/io/github/samera2022/chinese_chess/core/engine/GameEngine.java:509))

public Piece getPiece(int row, int col) {

### ✅ P0-2: Board 快照一致性保护

**文件**: [`ucc-core/.../Board.java`](../../ucc-core/src/main/java/io/github/samera2022/chinese_chess/core/engine/Board.java)

#### 正确实施的改动

| 改动 | 位置 | 评价 |
|------|------|------|
| `turn` 改为 `volatile` | :25 | 🟢 正确 |
| `deepCopy()` 改为 `toState()` → `fromState()` | :306-309 | 🟢 正确 |
| `toState()` 内 `synchronized (stacks)` 快照 keySet | :342-344 | 🟢 正确 |
| `toState()` 内 `synchronized (dq)` 复制每个堆栈 | :352-356 | 🟢 正确 |

#### 分析

`deepCopy()` 委托给 `BoardState` 快照的方案比原方案更简洁、更安全：
- 原方案: 遍历 `stacks` entrySet → 容易 `ConcurrentModificationException`
- 新方案: `toState()` 先快照 key → 再逐个复制 dq → 无并发修改风险

⚠️ 注意：`synchronized (dq)` 锁的是 Java 标准库的 `ArrayDeque` 对象。`ArrayDeque` 本身不是线程安全的，这里用 `synchronized` 块保护迭代过程。**正确的前提是**：所有修改 dq 的操作（`addLast`/`removeLast`/`clear`）也需要在同一锁保护下。当前 `Board` 内部操作（如 `pushToStack`、`popTop`）并未同步在 dq 上——但这些操作由 `GameEngine.engineLock` 保护，所以在 `GameEngine` 的使用场景下是安全的。如果将来有代码直接调用 `Board` 方法而不经过 `GameEngine` 锁，会出现问题。

### 🔴 新增发现 1: 持锁回调死锁风险

**位置**: 所有 `notifyGameStateChanged()` 调用点

[`GameEngine.java`](ucc-core/src/main/java/io/github/samera2022/chinese_chess/core/engine/GameEngine.java) 中 `notifyGameStateChanged()` 在以下场景被调用：
- `makeMove()` → `checkGameState()` → `notifyGameStateChanged()`（持锁 :130）
- `undoLastMove()` → `notifyGameStateChanged()`（持锁 :425）
- `restart()` → `notifyGameStateChanged()`（持锁 :474）
- `rebuildBoardForTopBottom()` → `notifyGameStateChanged()`（持锁 :96）
- `notifyListenersRefresh()` → `notifyGameStateChanged()`（持锁 :907）

```java
// makeMove() 内部 — 持有 engineLock
private void notifyGameStateChanged() {
    for (GameStateListener listener : listeners) {
        listener.onGameStateChanged(gameState);  // 外部回调!
    }
}
```

**风险**: 监听器（如 `GameController.onGameStateChanged()` → `updateStatus()`）内部调用 `session.getGameStatus()`、`session.isRedTurn()`、`session.getMoveHistory()` 等方法。由于 `ReentrantLock` 对同一线程可重入，**同一线程内的回调是安全的**。但如果监听器将任务 dispatch 到另一个线程（如 `SwingUtilities.invokeLater`），而那个线程又尝试获取锁，就会**阻塞**（不会死锁，但会等待）。

**当前实际影响**: 低。`GameController.onGameStateChanged()` 和 `onMoveExecuted()` 都使用 `SwingUtilities.invokeLater` → `updateStatus()`，而 `updateStatus()` 调用的是 `session.getGameStatus()`、`session.isRedTurn()`（均加锁）。EDT 线程获取锁时会阻塞直到 `makeMove()` 释放锁。这在人类点击频率下不可察觉，但在高并发服务端可能导致短暂排队。

**建议**: 将监听器通知移到锁外：
```java
engineLock.unlock();
try {
    // 通知监听器
    for (GameStateListener listener : listeners) {
        listener.onMoveExecuted(move);
    }
} finally {
    engineLock.lock(); // 如果需要继续持锁
}
```
或在文档中标注"监听器回调在 engineLock 持有期间执行，禁止回调内调用 GameEngine 方法（除 EDT dispatch 外）"。

### 🔴 新增发现 2: `AIModeEnabler` 跨线程 Board 竞态

**位置**: [`AIModeEnabler.java:146-150`](../../ucc-app/src/main/java/io/github/samera2022/chinese_chess/app/ui/AIModeEnabler.java:146)

```java
// AIModeEnabler.triggerAIMove() 运行在 aiExecutor 线程 (非 EDT)
ReadonlyBoard rb = session.getBoard();              // 获取 Board 引用 (通过锁)
// ... 锁已释放 ...
SimulationBoard simBoard = new SimulationBoard((Board) rb);  // 在无锁下拷贝!
```

**风险分析**:
1. `aiExecutor` 线程调用 `session.getBoard()` → 获得 `engineLock` → 拿到 `board` 引用 → 释放锁
2. 同一时刻，EDT 线程可能正在 `makeMove()`（持有 `engineLock`）中修改 `board`
3. `aiExecutor` 线程在**无锁**状态下构造 `SimulationBoard`，遍历 `source.getStack(r,c)`（返回副本 ArrayList，安全），但可能捕获到 EDT 修改**中间状态**的快照不一致

**实际影响**: 低。`getStack()` 返回 `new ArrayList<>(dq)` 副本，不会抛出异常。但 AI 可能基于一个不一致的局面（如某棋子刚被移除但回合尚未翻转）进行评估。

**建议**: `AIModeEnabler` 改用 `session.getBoardState()`（返回不可变快照 + 已加锁），再从 `Board.fromState()` 重建。

### 🔴 新增发现 3: `ForceMoveHandler` 反射 hack 脆弱性

**位置**: [`ForceMoveHandler.java:160-165`](../../ucc-app/src/main/java/io/github/samera2022/chinese_chess/app/ui/ForceMoveHandler.java:160)
