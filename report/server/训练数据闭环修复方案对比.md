# 训练数据闭环修复 — 方案 A vs 方案 B 对比评估

> 日期: 2026-06-01  
> 关联: [ucc-server实现审计报告_v3.md](ucc-server实现审计报告_v3.md)

---

## 问题回顾

当前 `startParallelGames()` 使用 `CompletableFuture.runAsync(worker)` 提交 Worker，Worker 引用丢失。100 个 Worker 各自收集的 `TrainingDataCollector` 样本在 `run()` 结束后随 GC 丢弃，`TrainingOrchestrator` 无法获取训练数据。

---

## 方案 A: `SelfPlayWorker implements Callable<TrainingDataCollector>`

**改动**:

```java
// SelfPlayWorker.java
-public class SelfPlayWorker implements Runnable {
+public class SelfPlayWorker implements Callable<TrainingDataCollector> {

-    @Override public void run() {
+    @Override public TrainingDataCollector call() {
         // ... 一模一样的主体逻辑 ...
+        return collector;
     }

// BatchingEngine.java
-    futures[i] = CompletableFuture.runAsync(worker, workerPool);
+    futures[i] = CompletableFuture.supplyAsync(
+        () -> { worker.call(); return worker.getCollector(); }, workerPool);
```

**问题**: `call()` 返回值与 `getCollector()` 返回**同一个对象**。要取数据还得回到 `future.get()` 或 `getCollector()`，`Callable<Collector>` 是增加的抽象层但无信息增量。

---

## 方案 B: `startParallelGames()` 返回 Worker 引用列表

**改动**: 新增轻量结果封装类，`startParallelGames()` 同时返回 Future + Worker 列表。

```java
// BatchingEngine.java — 新增 record
public record GameBatchResult(CompletableFuture<Void> future,
                               List<SelfPlayWorker> workers) {}

public GameBatchResult startParallelGames(int numWorkers, GameRulesConfig rules) {
    List<SelfPlayWorker> workers = new ArrayList<>();
    for (int i = 0; i < numWorkers; i++) {
        SelfPlayWorker worker = new SelfPlayWorker(i, rules, this);
        workers.add(worker);
        futures[i] = CompletableFuture.runAsync(worker, workerPool);
    }
    return new GameBatchResult(CompletableFuture.allOf(futures), workers);
}

// TrainingOrchestrator.java — 调用方
GameBatchResult result = batchingEngine.startParallelGames(count, rules);
result.future().join();
for (SelfPlayWorker w : result.workers()) {
    trainingData.addAll(w.getCollector().getSamples());
}
```

---

## 对比矩阵

| 维度 | 方案 A (`Callable`) | 方案 B (返回 Worker 列表) |
|------|:---|:---|
| 改动范围 | 3 个文件，~10 行 | 2 个文件，~20 行（含 4 行新 record） |
| `SelfPlayWorker` 需改接口签名 | ✅ 改 `implements Runnable` → `implements Callable` | ❌ 不改，保持 `Runnable` |
| 是否有冗余抽象 | ❌ `call()` 返回值 ≡ `getCollector()`，`Callable` 是多余的包装层 | ❌ 无冗余，Worker 引用直接暴露 |
| 与 `CompletableFuture` 兼容性 | ❌ `Callable` 非 `runAsync()` 一等公民，须包 `supplyAsync` | 🟢 `runAsync(Runnable)` 直接支持 |
| 语义正确性 | 🟡 "执行对局"是副作用操作，不应建模为"计算返回值" | 🟢 `Runnable` 准确表达副作用语义 |
| 调用方复杂度 | 🟢 `future.get()` 单次调用 | 🟡 遍历 `workers` 列表调用 `getCollector()` |
| 扩展性 | ❌ 未来加元数据需改 `call()` 返回值类型 | 🟢 Worker 引用在手，加 getter 即可 |
| 线程安全性 | 🟢 `collector` 在 `call()` 结束前不会被外部访问 | 🟢 调用方在 `future.join()` 之后才访问 worker |
| 向后兼容 | ❌ `startSelfPlay()` `@Deprecated` 旧接口也需同步改动 | 🟢 新增重载，旧接口可保留不变 |

---

## 建议: 方案 B

**四条决定性理由**:

1. **`Runnable` 是正确的语义**。"执行一局自博弈"的本质是副作用操作（改变棋盘状态、收集训练样本），不是计算一个值。用 `Callable` 把它建模成"返回值是 Collector"是在语义层面扭曲了操作意图。方案 B 保持 `Runnable`，副作用归副作用，数据容器归数据容器，职责分离清晰。

2. **`Callable` 在此场景下是纯开销抽象**。方案 A 落地后 `call()` 的返回值和 `getCollector()` 是完全相同的对象引用——增加了一层接口约束但没有带来任何抽象收益。如果未来 `call()` 要返回更多东西（如步数、终局原因），返回值类型需要改成复合 DTO，调用方也跟着改。

3. **方案 B 扩展路径更平滑**。Worker 引用暴露后，调用方可以按需扩展——今天取 `getCollector()`，明天取 `getMoveCount()`、`getGameDuration()`——不需要改接口签名、不需要改 `call()` 返回值、不需要改类型参数。加一个 getter 即可。

4. **改动更安全**。方案 A 必须改 `SelfPlayWorker` 的接口声明（`implements Runnable` → `implements Callable`），如果 IDE 全局重构不彻底会有遗漏。方案 B 只改 `BatchingEngine` 一个方法签名和 `TrainingOrchestrator` 一处调用，`SelfPlayWorker` 源码零改动。
