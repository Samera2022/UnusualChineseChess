# ucc-core P0 线程安全改造方案

> 版本: 1.0  
> 日期: 2026-06-01  
> 前置阅读: [ucc-core并发评估与解耦分析.md](../ucc-core并发评估与解耦分析.md)

---

## 一、改造目标

使 `ucc-core` 模块从**单线程（Swing EDT）设计**升级为**多线程安全设计**，支撑以下场景：

| 场景 | 并发模型 | 对 core 的要求 |
|------|----------|----------------|
| 网络对战（ucc-server） | 每房间一个 `GameEngine`，独立线程 | 单实例内线程安全 |
| 并行自我对弈训练 | 100+ Worker 线程各持 `SimulationBoard` | 各实例独立，`fork()` 隔离 |
| 批量推理（Batching Engine） | 多个 `Board.toState()` 并发调用 | `toState()`/`deepCopy()` 线程安全 |

### 核心原则

- **不引入重量级锁**：优先使用不可变快照（`BoardState`）、`CopyOnWriteArrayList`、`volatile`
- **保持向后兼容**：所有公共 API 签名不变
- **最小侵入**：仅修改必须改的类，不影响现有 GUI 逻辑

## 二、改造清单（按优先级排列）

### 🔴 P0-1: `GameEngine` 线程安全化

**文件**: [`ucc-core/.../GameEngine.java`](../../ucc-core/src/main/java/io/github/samera2022/chinese_chess/core/engine/GameEngine.java)

#### 问题
- `makeMove()`、`undoLastMove()`、`restart()`、`loadSyncState()` 无任何同步
- `listeners` 列表用普通 `ArrayList`，add/remove 与迭代存在竞态
- `providerListener` 回调来自 `RulesConfigProvider`，线程上下文不确定

#### 改造方案

```java
public class GameEngine implements GameStateAccessor, GameSession {
    // ── 改为可重入锁，保护所有状态修改操作 ──
    private final ReentrantLock engineLock = new ReentrantLock();

    // ── 监听器改为线程安全列表 ──
    private final List<GameStateListener> listeners = new CopyOnWriteArrayList<>();

    // ── 线程安全的 sessionListenerMap ──
    private final ConcurrentHashMap<SessionListener, GameStateListener> sessionListenerMap
        = new ConcurrentHashMap<>();

    public boolean makeMove(int fromRow, int fromCol, int toRow, int toCol,
                            Piece.Type promotionType, int selectedStackIndex) {
        engineLock.lock();
        try {
            // ... 原有逻辑不变 ...
        } finally {
            engineLock.unlock();
        }
    }

    public boolean undoLastMove() {
        engineLock.lock();
        try {
            // ... 原有逻辑不变 ...
        } finally {
            engineLock.unlock();
        }
    }

    public void restart() {
        engineLock.lock();
        try {
            // ... 原有逻辑不变 ...
        } finally {
            engineLock.unlock();
        }
    }

    public void loadSyncState(JsonObject root) {
        engineLock.lock();
        try {
            // ... 原有逻辑不变 ...
        } finally {
            engineLock.unlock();
        }
    }

    // ── 读方法也加锁（保证读到一致状态）──
    public BoardState getBoardState() {
        engineLock.lock();
        try {
            return board.toState();
        } finally {
            engineLock.unlock();
        }
    }

    public GameStatus getGameStatus() {
        engineLock.lock();
        try {
            return gameState;
        } finally {
            engineLock.unlock();
        }
    }

    // ── addGameStateListener/removeGameStateListener 无需加锁 ──
    // CopyOnWriteArrayList 自身线程安全

    // ── providerListener 回调加锁保护 ──
    private final RulesConfigProvider.InstanceChangeListener providerListener = (oldInst, newInst) -> {
        engineLock.lock();
        try {
            this.rulesConfig = newInst;
            if (this.validator != null) this.validator.setRulesConfig(newInst);
        } finally {
            engineLock.unlock();
        }
    };
}
```

#### 改动量估算
- 约 20 个方法加锁包裹
- 3 个集合类型替换
- **零公共 API 变更**

### 🔴 P0-2: `Board` 快照一致性保护

**文件**: [`ucc-core/.../Board.java`](../../ucc-core/src/main/java/io/github/samera2022/chinese_chess/core/engine/Board.java)

#### 问题
- `deepCopy()` 遍历 `stacks` 的 `entrySet()`，并发 `pushToStack`/`popTop` 会抛 `ConcurrentModificationException`
- `toState()` 同理
- `turn` 字段包级私有（被 `SimulationBoard` 直接访问），无 `volatile`

#### 改造方案

**方案**: 将 `deepCopy()` 和 `toState()` 改为基于 `BoardState` 中间快照，避免直接遍历可变集合。

```java
public Board deepCopy() {
    // 通过不可变快照构造副本，避免遍历可变 Map
    BoardState snapshot = this.toState();      // toState 内部已做快照
    return Board.fromState(snapshot);
}

@Override
public BoardState toState() {
    List<BoardState.StackEntry> entries = new ArrayList<>();
    // 先复制 keySet 再遍历，避免并发修改
    String[] keys;
    synchronized (stacks) {
        keys = stacks.keySet().toArray(new String[0]);
    }
    for (String key : keys) {
        Deque<Piece> dq = stacks.get(key);  // 可能已被 remove，判空
        if (dq == null || dq.isEmpty()) continue;
        String[] parts = key.split(",");
        int r = Integer.parseInt(parts[0]);
        int c = Integer.parseInt(parts[1]);
        List<Piece.Type> types = new ArrayList<>();
        // 复制 Deque 内容
        synchronized (dq) {
            for (Piece p : dq) {
                types.add(p.getType());
            }
        }
        entries.add(new BoardState.StackEntry(r, c, types));
    }
    return new BoardState(rows, COLS, entries, turn);
}
```

**备选方案（更简洁但侵入更大）**: 将 `stacks` 改为 `ConcurrentHashMap<String, Deque<Piece>>`，每个 `Deque` 改为 `ConcurrentLinkedDeque`。但这会改变大量内部操作逻辑。**当前不推荐**。

#### 改动量估算
- `deepCopy()` 重写为委托调用：3 行
- `toState()` 增加同步块：约 10 行
- `turn` 加 `volatile`：1 行

### 🔴 P0-3: `GameRulesConfig` 线程池泄漏修复

**文件**: [`ucc-core/.../GameRulesConfig.java`](../../ucc-core/src/main/java/io/github/samera2022/chinese_chess/core/rules/GameRulesConfig.java)

#### 问题
每个 `GameRulesConfig` 实例创建 2 个 `ExecutorService`（守护线程池）。高并发训练中若频繁创建实例，线程泄漏严重。

#### 改造方案

**将线程池改为静态共享**：

```java
public class GameRulesConfig {
    // ── 改为静态共享线程池 ──
    private static final ExecutorService NOTIFY_EXECUTOR = Executors.newSingleThreadExecutor(r -> {
        Thread t = new Thread(r, "RuleChangeNotifier");
        t.setDaemon(true);
        return t;
    });
    private static final ExecutorService LISTENER_EXECUTOR = Executors.newSingleThreadExecutor(r -> {
        Thread t = new Thread(r, "RuleChangeListener");
        t.setDaemon(true);
        return t;
    });

    // ── shutdown() 改为静态方法，全局仅调用一次 ──
    public static void shutdownExecutors() {
        NOTIFY_EXECUTOR.shutdownNow();
        LISTENER_EXECUTOR.shutdownNow();
    }

    // ── 实例级 shutdown() 改为空操作（或仅清理监听器）──
    public void shutdown() {
        // 不再关闭线程池，仅清理实例级资源
        changeListeners.clear();
    }
}
```

**同时**：`RulesConfigProvider.shutdown()` 调用新的静态方法。

#### 改动量估算
- 2 个字段改为 `static`
- 修改 `shutdown()` 逻辑
- 新增 `shutdownExecutors()` 静态方法

### 🟡 P1-1: `CheckDetector.canEscapeCheck()` 无副作用化

**文件**: [`ucc-core/.../CheckDetector.java`](../../ucc-core/src/main/java/io/github/samera2022/chinese_chess/core/rules/CheckDetector.java)

#### 问题
`canEscapeCheck()` 直接在原棋盘上执行 `removePiece` → `pushToStack` → `clearStack` 再恢复，多线程共享时会互相干扰。

#### 改造方案

**改为在 SimulationBoard 副本上操作**：

```java
private boolean canEscapeCheck(Piece piece, int toRow, int toCol, boolean isRed) {
    // 创建轻量级副本进行试探（SimulationBoard 构造即深拷贝）
    SimulationBoard simBoard = new SimulationBoard(board);
    // 在副本上操作对应棋子
    Piece simPiece = simBoard.getPiece(piece.getRow(), piece.getCol());
    if (simPiece == null) return false;

    // 执行试探走子
    simBoard.removePiece(simPiece.getRow(), simPiece.getCol());
    simBoard.clearStack(toRow, toCol);        // 清除目标位置
    simPiece.move(toRow, toCol);
    simBoard.pushToStack(toRow, toCol, simPiece);

    // 在副本上检查
    CheckDetector simDetector = new CheckDetector(simBoard, new MoveValidator(simBoard));
    return !simDetector.isInCheck(isRed);
}
```

#### 改动量估算
- `canEscapeCheck()` 方法重写：约 15 行
- 性能影响：`isCheckmate()` 在将军时遍历所有合法走法，对每个走法都 `new SimulationBoard()`。粗略估算：平均 30 合法走法 × 每局 Max 200 步 = 6000 次 copy。`SimulationBoard` 构造包含全盘深拷贝（18×9 + 堆栈），约 0.1~0.3ms/次，单局额外开销约 0.6~1.8s。

**优化**：可以缓存 `MoveValidator` 实例，用后即弃。

### 🟡 P1-2: JDK 版本升级（Java 11 → Java 21）

#### 动机
- 利用**虚拟线程 (Virtual Threads)** 管理大量并发对局
- `ReentrantLock` 在虚拟线程下自动 pin 载体线程的问题已有解决方案
- 更好的 GC（ZGC 在 JDK 21 中已生产就绪）

#### 改动
- 根 `pom.xml`：`<maven.compiler.source>21</maven.compiler.source>`
- 确保所有依赖兼容 JDK 21（Gson 2.11.0 已兼容）
- 现有的 `Thread` 创建可逐步迁移到 `Thread.ofVirtual()`

#### 风险
- `ucc-app` (Swing GUI) 需验证 JDK 21 兼容性
- 构建脚本 (`build-local.ps1`) 需更新 JDK 路径

## 三、改造后的线程安全模型

```
┌──────────────────────────────────────────────────────┐
│                    ucc-server                        │
│  ┌──────────────────────────────────────────────┐   │
│  │  Room 1 (Virtual Thread)                     │   │
│  │  ┌──────────┐  ┌──────────┐                 │   │
│  │  │GameEngine│  │  Board   │                 │   │
│  │  │Reentrant │  │ toState()│                 │   │
│  │  │  Lock    │  │synchronized               │   │
│  │  └──────────┘  └──────────┘                 │   │
│  └──────────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────────┐   │
│  │  Room N (Virtual Thread) — 同上，完全隔离      │   │
│  └──────────────────────────────────────────────┘   │
│                                                      │
│  ┌──────────────────────────────────────────────┐   │
│  │  GameRulesConfig (共享单例)                    │   │
│  │  synchronized + 静态线程池                     │   │
│  └──────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────┘
```

**核心保证**：
1. 不同 `GameEngine` 实例之间**完全隔离**（各持独立 `Board`）
2. 同一 `GameEngine` 内所有状态修改由 `ReentrantLock` 串行化
3. 共享的 `GameRulesConfig` 通过 `synchronized` 保护
4. `Board.toState()` / `Board.deepCopy()` 通过快照机制避免并发异常

## 五、新增测试用例

### 5.1 `GameEngineConcurrencyTest.java`

```java
@Test
public void testConcurrentMakeMoveAndUndo() throws Exception {
    GameEngine engine = new GameEngine();
    int threads = 10;
    ExecutorService exec = Executors.newFixedThreadPool(threads);
    CyclicBarrier barrier = new CyclicBarrier(threads);

    for (int i = 0; i < threads; i++) {
        exec.submit(() -> {
            try {
                barrier.await();
                for (int j = 0; j < 100; j++) {
                    engine.getBoardState();  // 读
                    engine.makeMove(9, 0, 8, 0);  // 写（可能因回合不对而返回 false）
                    engine.getGameStatus();
                }
            } catch (Exception e) {
                fail(e.getMessage());
            }
        });
    }
    exec.shutdown();
    assertTrue(exec.awaitTermination(30, TimeUnit.SECONDS));
}
```
