## 2. Phase 0: 规则感知验证实验（最优先）

> **核心理念**: 在投入大规模算力之前，先用最小实验验证"规则向量输入是否真的改变 AI 走法"。

### 2.1 实验目标

**验证假设**: 给定同一个残局局面、不同的规则向量，AI 的走法在至少 10% 的测试样本上产生差异。

### 2.2 实验设计

```
输入:
  - 100 个固定残局局面（涵盖中局、残局）
  - 对每个局面，使用 2 组规则向量:
    A 组: 标准规则 (全部 false)
    B 组: 随机开启 1-3 条扩展规则

网络:
  - 最小 ResNet: 3 个残差块, 64 通道
  - 输入: 棋盘 14×10×9 (14 通道 = 7 种棋子 × 2 方)
          拼接 22 位规则向量 → 展平输入
  - 输出: 策略头 (90 路) + 价值头 (标量)

MCTS:
  - 每步模拟 400 次
  - 温度参数 τ=1 (鼓励探索)

指标:
  - 走法差异率: A 组和 B 组 top-1 着法不同的局面数 / 总局面数
  - 如果 < 10%: 规则嵌入无效，需要重新设计架构
  - 如果 > 10%: 验证通过，可以推进 Phase 1+
```

### 2.3 硬件需求

| 项目 | 需求 |
|------|------|
| GPU | 不需要，CPU 即可（MCTS 400 次 × 100 残局 × 2 规则组 = 80,000 次搜索） |
| 内存 | 4GB |
| 时间 | ~2小时的 Python 脚本运行 |

### 2.4 关键代码骨架

```python
# selfplay.py — 最小验证实验，纯 Python + subprocess 调 Java
import subprocess, json

def query_java_engine(board_state, rule_vector, move):
```

## 3. Phase 1: 补齐基础设施——SimulationBoard

### 3.1 为什么需要 `SimulationBoard`

[`SimulationContext`](ucc-common/src/main/java/io/github/samera2022/chinese_chess/common/spi/SimulationContext.java:6) SPI 已定义，但 ucc-core 中没有实现类。AI 的 MCTS 搜索需要在**不修改真实棋盘**的前提下快速模拟走子、评估结果、撤销走子。

当前 `Board.deepCopy()` 可以作为模拟的基础，但每次全量深拷贝太慢（包含所有 `Deque` 和 `Map` 的重建）。`SimulationBoard` 需要在 `Board` 基础上增加增量撤消能力。

### 3.2 设计方案

```java
// 位置: ucc-core/src/.../core/engine/SimulationBoard.java
package io.github.samera2022.chinese_chess.core.engine;

import io.github.samera2022.chinese_chess.common.model.Move;
import io.github.samera2022.chinese_chess.common.model.Piece;
import io.github.samera2022.chinese_chess.common.spi.SimulationContext;
import io.github.samera2022.chinese_chess.common.spi.ReadonlyBoard;
import java.util.*;

public class SimulationBoard extends Board implements SimulationContext {

    // 增量撤消栈：记录每步模拟走子前的状态，避免每次 deepCopy()
    private final Deque<UndoRecord> undoStack = new ArrayDeque<>();

    private static class UndoRecord {
        final int fromRow, fromCol, toRow, toCol;
        final Piece movedPiece;
        final Piece capturedTop; // 被吃/堆叠的目标位置原有棋子
        final List<Piece> capturedStack; // 目标位置原有堆栈
    }

    public SimulationBoard(Board source) {
        // 从源棋盘初始化
        for (int r = 0; r < source.getRows(); r++)
            for (int c = 0; c < source.getCols(); c++)
                for (Piece p : source.getStack(r, c))
                    pushToStack(r, c, new Piece(p.getType(), r, c));
    }

    @Override
    public boolean simulateMove(int fromRow, int fromCol, int toRow, int toCol) {
        Piece piece = getPiece(fromRow, fromCol);
        if (piece == null) return false;

        // 保存撤消记录
        Piece captured = getPiece(toRow, toCol);
        List<Piece> capturedStack = captured != null
            ? new ArrayList<>(getStack(toRow, toCol)) : null;
        undoStack.push(new UndoRecord(fromRow, fromCol, toRow, toCol,
            piece, captured, capturedStack));

        // 执行走子 (复用 Board 的 removePiece/pushToStack)
        removePiece(fromRow, fromCol);
        piece.move(toRow, toCol);
        setPiece(toRow, toCol, piece);

        // 翻转回合
        flipTurn();
        return true;
    }

    @Override
    public boolean simulateUndo() {
        if (undoStack.isEmpty()) return false;
        UndoRecord rec = undoStack.pop();

        // 恢复移动的棋子
        removePiece(rec.toRow, rec.toCol);
        rec.movedPiece.move(rec.fromRow, rec.fromCol);
        pushToStack(rec.fromRow, rec.fromCol, rec.movedPiece);

        // 恢复被吃/堆叠的棋子
        if (rec.capturedStack != null) {
            for (Piece p : rec.capturedStack) {
                pushToStack(rec.toRow, rec.toCol,
                    new Piece(p.getType(), rec.toRow, rec.toCol));
            }
        }
        flipTurn();
        return true;
    }

    @Override
    public SimulationContext fork() {
        // 用于并行 MCTS: 深拷贝当前状态
        SimulationBoard copy = new SimulationBoard(this);
        copy.turn = this.turn;
        return copy;
    }

    @Override
    public int evaluate() {
        // 简单启发式评估：子力优势 + 位置优势
        // 后续会被神经网络替代
        return heuristicEval();
    }

    @Override
    public List<Move> generateLegalMoves() {
        List<Move> legal = new ArrayList<>();
        for (int r = 0; r < getRows(); r++) {
            for (int c = 0; c < getCols(); c++) {
                Piece p = getPiece(r, c);
                if (p == null || p.isRed() != isRedTurn()) continue;
                for (int tr = 0; tr < getRows(); tr++) {
                    for (int tc = 0; tc < getCols(); tc++) {
                        if (isValidMove(r, c, tr, tc)) {
                            legal.add(new Move(r, c, tr, tc, p, getPiece(tr, tc)));
                        }
                    }
                }
            }
        }
        return legal;
    }

    // ... 其他辅助方法
}
```

### 4.2 规则向量的使用方式

```
棋盘表示:  [14 × 10 × 9]  (14 通道: 每方 7 种棋子)  →  CNN 编码 → 局面向量 [256]
规则向量:  [23]           (22 布尔 + 1 连续)          →  全连接层 → 规则向量 [64]

拼接: [256 + 64] = [320]  →  策略头 + 价值头
```

**注入方式**: 在网络的瓶颈层（bottleneck）进行拼接，而非输入层。原因：

1. 输入层拼接会让规则向量在前几层被"淹没"
2. 瓶颈层拼接让规则信息直接引导高层决策

## 5. Phase 3: 神经网络架构设计

### 5.1 MVP 架构（Phase 0-1）：Mini ResNet

```python
# model.py
import torch
import torch.nn as nn

class MiniResNet(nn.Module):
    """最小可行 ResNet — 用于规则感知验证实验"""

    def __init__(self, board_channels=14, board_h=10, board_w=9,
                 rule_dim=23, num_res_blocks=3, filters=64):
        super().__init__()

        # 棋盘编码器
        self.board_conv = nn.Sequential(
            nn.Conv2d(board_channels, filters, 3, padding=1),
            nn.BatchNorm2d(filters),
            nn.ReLU()
        )
        self.res_blocks = nn.Sequential(*[
            ResidualBlock(filters) for _ in range(num_res_blocks)
        ])

        # 规则编码器
        self.rule_fc = nn.Sequential(
            nn.Linear(rule_dim, 64),
            nn.ReLU()
        )

        # 策略头 (输出: W×H 路的概率分布)
        self.policy_conv = nn.Conv2d(filters, 2, 1)
        self.policy_fc = nn.Linear(2 * board_h * board_w + 64,
                                    board_h * board_w)

        # 价值头 (输出: [-1, 1] 标量)
        self.value_conv = nn.Conv2d(filters, 1, 1)
        self.value_fc = nn.Sequential(
            nn.Linear(board_h * board_w + 64, 256),
            nn.ReLU(),
            nn.Linear(256, 1),
            nn.Tanh()
        )

    def forward(self, board, rule_vec):
        # 棋盘特征提取
        x = self.board_conv(board)
        x = self.res_blocks(x)  # [B, 64, 10, 9]

        # 规则编码
        r = self.rule_fc(rule_vec)  # [B, 64]

        # 策略: 拼接规则向量到瓶颈层
        p = self.policy_conv(x)  # [B, 2, 10, 9]
        p = p.flatten(1)          # [B, 180]
        p = torch.cat([p, r], dim=1)  # [B, 244]
        policy = self.policy_fc(p)    # [B, 90] — 每格一个概率

        # 价值
        v = self.value_conv(x)   # [B, 1, 10, 9]
        v = v.flatten(1)          # [B, 90]
        v = torch.cat([v, r], dim=1)  # [B, 154]
        value = self.value_fc(v)  # [B, 1]

        return policy, value
```

## 7. Phase 5: 训练流水线

### 7.1 训练流程

```
1. 初始化模型 (随机权重)
2. 循环 N 代:
   a. 自我对弈 (Self-Play):
      - 随机采样规则向量
      - 用当前模型 + MCTS (400 次模拟) 生成对弈棋谱
      - 保存 (局面, 规则向量, MCTS 策略, 最终胜负) 到 replay buffer
   b. 训练 (Training):
      - 从 replay buffer 随机采样 batch
      - 损失 = CrossEntropy(策略) + MSE(价值)
      - 反向传播，更新模型权重
   c. 评估 (Evaluation):
      - 新模型 vs 旧模型，对战 100 局
      - 如果新模型胜率 > 55%，替换旧模型
3. 导出 model.pt (TorchScript)
```

### 7.2 课程学习策略

```
第 1 阶段 (10% 训练时间): 仅标准规则
  规则向量 = [0,0,0,...,0]
  目标: 学会基本象棋走法

第 2 阶段 (40% 训练时间): 引入扩展规则 (outside + extended)
  规则向量 = 3 位基础 + 14 位延申 随机开启
  目标: 学会规则感知
```

### 8.1 推理流程

```java
// ucc-ai/src/.../ai/RuleAwareAI.java
public class RuleAwareAI implements AIStrategy {

    private final Model model; // TorchScript 加载的模型

    @Override
    public Move findBestMove(SimulationContext ctx, int maxDepth, long timeLimitMs) {
        ReadonlyBoard board = ctx.getBoard();
        float[] ruleVec = RuleEncoder.encode(getCurrentConfig());
        List<Move> legalMoves = ctx.generateLegalMoves();

        // MCTS 搜索
        MCTSNode root = new MCTSNode(board, ruleVec, null, null);
        long deadline = System.currentTimeMillis() + timeLimitMs;
        while (System.currentTimeMillis() < deadline) {
            MCTSNode leaf = root.select();     // UCB 选择
            if (!leaf.isTerminal()) {
                float value = model.evaluate(leaf.board, leaf.ruleVec); // 神经网络评估
                leaf.expand(value);
            }
            leaf.backpropagate();
        }

        // 选择访问次数最多的着法
        return root.bestMove();
    }

    @Override public String getName() { return "RuleAware-RL"; }
    @Override public AIStrategyConfig getConfig() { ... }
    @Override public void applyConfig(AIStrategyConfig config) { ... }
}
```

## 附录 A: Rust 引擎复刻策略分析

### A.1 为什么需要 Rust 引擎

MCTS 自我对弈的性能瓶颈在规则引擎。每步模拟需要调用数百至数千次 `isValidMove()` / `simulateMove()` / `simulateUndo()`。当前纯 Java 引擎（[`MoveValidator`](ucc-core/src/main/java/io/github/samera2022/chinese_chess/core/rules/MoveValidator.java:7) + [`CheckDetector`](ucc-core/src/main/java/io/github/samera2022/chinese_chess/core/rules/CheckDetector.java:9) + [`Board`](ucc-core/src/main/java/io/github/samera2022/chinese_chess/core/engine/Board.java:14)）在效率上存在三个障碍：

| 障碍 | 根因 | 对 AI 训练的影响 |
|------|------|-----------------|
| GC 抖动 | 每步模拟产生大量临时 `Piece`/`Deque`/`ArrayList` 对象，GC 暂停打断 MCTS 展开 | 吞吐波动大，训练不稳定 |
| 虚方法调用开销 | `Board.getPiece()` → `stacks.get(key)` → `Deque.peekLast()` 三次间接跳转 | 单步模拟约 1μs，MCTS 10000 次 = 10ms/步 |
| 堆栈语义过重 | 当前的 `Map<String, Deque<Piece>>` 用 HashMap 索引位置，走子时多次拆/装 Deque | 与位棋盘（Bitboard）方案相差 5-20 倍 |

### A.2 三种路线对比

#### 路线 1: Java SimulationBoard（已在 Phase 1 中定义）

```
工作量: 1 天（纯 Java，直接扩展 Board）
与现有系统集成: 天然集成，零成本
对 AI 训练的提升: 基线（无提升）
风险: 无
```

#### 路线 2（推荐）: Rust 独立复刻——仅用于训练

```
┌─────────────────────────────────────────────┐
│  Python 训练脚本 (PyTorch)                   │
│    │                                         │
│    │  Python ctypes / PyO3 调用               │
│    ▼                                         │
│  librules.so (Rust 编译的共享库)              │
│    ├── Board (位棋盘实现，u64×10 行)           │
│    ├── MoveValidator (纯函数，无 GC)          │
│    ├── CheckDetector (同样纯函数)              │
│    └── SimulationContext (栈上模拟，无需 heap) │
│                                              │
│  ⚠ 不接入 Java 系统！仅 Python 调用！          │
│  无需改动任何 Java 代码                        │
└─────────────────────────────────────────────┘

工作量: 2-3 周
  - 翻译 MoveValidator 的 7 种棋子走法校验逻辑 (~200 行 Rust)
  - 翻译 CheckDetector 的将军/将死检测 (~60 行 Rust)
  - 翻译 Board 的棋子状态管理，使用位棋盘替代 HashMap (~150 行 Rust)
  - Python FFI 绑定 (~80 行 Rust + ~50 行 Python)
  - 一致性测试：1000 个随机残局，Rust vs Java，每一步模拟结果必须一致

与现有系统集成: 不需要集成
对 AI 训练的提升: 预计 5-20 倍吞吐提升（无 GC + 位棋盘 + 纯函数）
风险: 中（逻辑翻译错误可能导致训练数据偏差，需严格一致性测试）
共享内容: Rust 和 Java 共享同一套"规则逻辑"（26 条 RuleRegistry 规则）
          但各写各的实现
```

#### 路线 3（可选）: Rust 完全替代——同时接入 Java 系统

```
工作量: 4-6 周（在路线 2 基础上追加）
  - JNI 绑定层 (~200 行 Rust + ~100 行 Java)
  - ucc-core 中新增 RustEngine 类，实现 GameSession + GameStateAccessor
  - 适配 RulesConfigProvider 的实例切换机制
  - 适配 GameRulesConfig.ChangeSource 的变化通知 → Rust 端

对现有系统的影响: 大
  - ucc-app 的所有组件目前通过 GameSession SPI 接口访问引擎，RustEngine
    只需实现相同接口即可无缝替换。BoardPanel/GameController/etc 不需改动。
  - 但 ChineseChessFrame 的构造函数需要增加"选择引擎"的分支逻辑
  - RulesConfigProvider 的全局单例需要改造为"可注入"以支持 Rust 侧规则同步

对 AI 训练的提升: 同路线 2（训练时就用 librules.so）
对游戏运行时的提升: 本地单人模式不影响（人类玩家感受不到 GC 差异）
                    联机模式中服务端可用 Rust 引擎加速

风险: 高——JNI 内存管理、跨语言规则同步、异常处理、调试复杂度
```

### A.3 推荐决策

```
Phase 0-2 期间: 使用 Java SimulationBoard（路线 1）
  → 先验证规则感知 + MCTS 收敛
  → 确认"AI 能下棋"之后再考虑加速

Phase 3 开始: 并行开发 Rust 独立复刻（路线 2）
  → Python→Rust FFI，不需要触碰 Java 代码
  → 用 Java 版本做"黄金标准"逐条验证 Rust 实现
  → 出任何 bug 只影响训练数据，不影响线上游戏

Phase 4+ (远期): 视需要评估路线 3
  → 仅当训练成熟、需要线上部署 AI 时考虑
  → 此时 GameSession SPI 的可替换性会体现价值
```

### A.4 Rust 复刻（路线 2）的详细规格

#### 核心类型映射

| Java 类型 | Rust 类型 | 说明 |
|-----------|----------|------|
| `Piece.Type` (14 种枚举) | `#[repr(u8)] enum PieceType` | 1 字节表示 |
| `Board` (10×9 / 18×9) | `struct Board { cells: [Option<(PieceType, u8)>; N]; ... }` | 用 Option + stack_depth 表示堆叠 |
| `MoveValidator` | `fn is_valid_move(board: &Board, rules: &RuleConfig, ...) -> bool` | 纯函数，无状态 |
| `GameRulesConfig` | `struct RuleConfig { bits: u32 }` | 22 个规则用 32 位位掩码 |
| `Move` | `struct Move { from: u8, to: u8, ... }` | 压缩至 4 字节 |

#### 位棋盘设计（关键加速点）

```
// 每种棋子类型 x 每方 = 14 个 u128 位棋盘
// 10 行 × 9 列 = 90 个位置，u128 的 90 位刚好覆盖
struct BitBoards {
    red_king:    u128,
    red_advisor: u128,
    red_elephant: u128,
    red_horse:   u128,
    red_chariot: u128,
    red_cannon:  u128,
    red_soldier: u128,
    // 黑方同理
    black_king:    u128,
    // ...
    occupied:     u128,  // 所有棋子 OR
    red_occupied: u128,
    black_occupied: u128,
}
```

优势：
- 车炮直线行走只需位运算查 `occupied` 掩码
- 马腿/象眼检查只需 `bitboard & mask != 0`
- 无 HashMap、无 Deque、无堆分配

#### 堆叠棋子的处理（本项目独有特性）

堆叠（`ALLOW_PIECE_STACKING`）无法直接用位棋盘表示多层，需要一个附加数组：

```rust
// 每个位置最多 MAX_STACK 层
// 用 [Option<PieceType>; MAX_STACK] 只对实际有堆叠的位置分配
struct StackEntry {
    pieces: ArrayVec<PieceType, 16>,  // 最多 16 层堆叠
}

// 仅在 allow_piece_stacking 规则启用时使用
// 标准规则下（无堆叠）：纯位棋盘，零额外开销
```
